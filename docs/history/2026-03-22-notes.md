## Why a new session feels unfamiliar

A new Codex session can re-read the repo, but it does not inherit the current thread's implicit working set:
- rejected approaches
- verified assumptions
- current blockers
- preferred terminology and boundaries
- short-term reasoning trajectory

The raw `.codex/sessions/*.jsonl` files are event logs, not directly reusable working memory.

## What exists locally in `.codex`

Observed storage layers:
- `sessions/*.jsonl`: full rollout event streams
- `archived_sessions/*.jsonl`: archived event streams
- `history.jsonl`: user input history
- `state_5.sqlite`: runtime thread state, jobs, thread metadata, dynamic tools
- `logs_1.sqlite`: structured runtime logs
- `sqlite/codex-dev.db`: automations / inbox product data
- `.codex-global-state.json`: UI/product preferences
- `config.toml`: product configuration

Conclusion: there is plenty of raw state, but little explicit project handoff state.

## Why not just replay the whole session

Problems with replaying raw session logs:
- too large
- too noisy
- much of the value is implicit and transient
- expensive to inject into a fresh session
- not stable enough as a "project state" representation

Conclusion: the missing primitive is explicit working memory, not more transcript access.

## What `claude-peers-mcp` solves and what it does not

Repository studied:
- https://github.com/louislva/claude-peers-mcp

It provides:
- peer discovery across Claude Code instances on one machine
- text messaging between instances
- lightweight summaries

It does not provide:
- shared session cognition
- structured handoff
- shared task tree
- decision memory
- failure memory

Conclusion: useful as a message bus, insufficient as a memory system.

## OpenViking takeaways

Repository studied:
- https://github.com/volcengine/OpenViking

Important ideas worth borrowing:
- session as a first-class object
- explicit `commit()` phase for session -> memory extraction
- layered context representation: L0/L1/L2
- memory is not transcript; memory is extracted structured context
- retrieval is only one layer in a larger context system

What seems too heavy for current needs:
- full context database scope
- generalized memory/resource/skill platform
- full AGFS + vector index platform design
- broader product concerns beyond project memory for coding agents

Conclusion: a narrower, coding-first, project-scoped variant is more appropriate.

## RAG is not the silver bullet

RAG helps retrieve relevant text.
It does not decide:
- what should become memory
- what matters now
- what should be handed to the next session
- which old conclusions are obsolete

Recommended sequence:
1. define memory schema
2. store structured project memory
3. use SQLite + FTS / structured filters first
4. add embeddings later if recall is insufficient
5. add reranking only if recall is fine but ranking is poor

## Candidate project-memory categories for coding work

Better categories than generic assistant memory:
- decision
- failure
- constraint
- task
- artifact
- handoff
- pattern

## Practical design direction so far

A project memory system should likely have these layers:
- raw event store
- extraction / summarization
- project memory store
- retrieval + injection

Likely first implementation:
- repo-local markdown snapshots for commit-like handoff
- SQLite metadata + FTS index
- git-aware linking to branch / sha / file paths
- fresh session reads the latest handoff snapshot first, not the entire transcript

## Near-term design questions

Still open:
- exact schema for project memory items
- when to trigger session snapshot / commit
- whether snapshots are manual, automatic, or hybrid
- how to bind memory items to git commits / branches / files
- how to support side-session review without polluting the mainline

## Memos takeaways

Repository studied:
- https://github.com/usememos/memos

Observed shape:
- primarily a self-hosted quick-capture / note-taking system
- core objects are memos, relations, attachments, inbox, reactions, and sharing
- MCP support exposes note CRUD and search primitives
- content model is flat compared with session-aware memory systems

Useful ideas:
- fast manual capture
- simple durable markdown-native knowledge base
- MCP-readable external note store
- lightweight self-hosted deployment

What it does not solve directly:
- session commit semantics
- project-scoped handoff state
- memory lifecycle / hotness
- git-aware project history binding
- coding-agent-specific memory schema

Conclusion: valuable as a capture surface or external note base, but not sufficient as the architecture template for project memory.

## OpenClaw memory takeaways

Repository studied:
- https://github.com/openclaw/openclaw

Observed shape:
- memory source of truth is plain Markdown in the agent workspace
- default memory layout is `MEMORY.md` plus `memory/YYYY-MM-DD.md`
- memory is pluginized; default slot uses `memory-core`
- memory tools are deliberately narrow: `memory_search` and `memory_get`
- semantic memory is an enhancement layer over files, not the canonical store
- context assembly is separately pluginized via the context engine

Important ideas worth borrowing:
- files as source of truth
- explicit distinction between memory system and context engine
- memory plugin can stay narrow while context assembly evolves independently
- semantic recall over files is useful, but should remain secondary to the written memory itself
- optional backend swapping (`builtin SQLite` vs `QMD`) behind one search interface
- session transcripts can also be exported/indexed as a separate recall source

What seems especially relevant for project memory:
- long-term curated memory and daily working memory are split
- automatic pre-compaction memory flush reminds the model to write durable facts before context loss
- hybrid retrieval includes semantic search plus keyword search, MMR, and temporal decay
- memory can stay local-first without becoming a giant generalized platform

What it still does not fully solve for our specific use case:
- coding-project-specific memory schema is still not the core abstraction
- git-aware decision binding is not the central design
- project handoff between coding sessions is not the main product surface
- memory items are still file-oriented notes rather than structured project state objects

Conclusion: OpenClaw is the strongest reference so far for a pragmatic coding-agent memory direction.
It suggests a good architecture split:
- source of truth = workspace files
- recall layer = semantic + keyword retrieval
- context engine = separate assembly/compaction lifecycle
For our project, this looks more directly adaptable than OpenViking.

# Constitution

## Product Boundaries

- The product is a coding control system built on a file-first memory
  substrate.
- The canonical source of truth is visible project files, not hidden runtime
  state.
- The system is project-agnostic. Repo-specific governance may appear as
  evidence, not as the global ontology.

## Architecture Invariants

- Markdown files remain canonical.
- SQLite plus FTS5 is the phase-1 accelerator, not the authority layer.
- Active objective, pivot lineage, and exception-contract state must remain explicit.
- Context assembly must compile through active control state before historical
  recall.

## Quality Bar

- New concepts must have clear owner-layer semantics.
- Temporary deviation must be labeled as an exception contract, not merged into target
  architecture.
- Freshness and provenance must be surfaced when state may be stale.

## Validation Rules

- Important claims need observable evidence, not only narration.
- Schema changes should be pressure-tested on real project samples.
- Retrieval improvements do not count unless they improve real orientation or
  control quality.

## Forbidden Shortcuts

- Do not silently overwrite old objectives without a pivot record.
- Do not let project-specific workflow fields become mandatory global schema.
- Do not treat stale handoff packets as current truth.

## Audit Hooks

- current_task_mentions_active_objective
- current_task_mentions_active_round
- guarded_exception_paths_require_active_contract
